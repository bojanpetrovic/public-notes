1 Notes

1.1 Repo pattern

In Ecto, nothing happens unless you call method on Repo module. Unlike active records, Ecto is explicit about everything.

1.2 Packages

package usage
ecto core, which we can use without database
ectosql needed when workig with relationtal databases

When we need ecto_sql, we don’t have to explicitly require for ecto package because it is a dependency on ecto_sql package.

1.3 Pin operator

We can use pin operator (^) to use expressions in queries: 

q = from "artists", where: [name: ^artist_name], select: [:id, :name]
q = from "artists", where: [name: ^("Bill" <> "Evans"], select: [:id, :name]

Any value added by pin operator become parametrized value. That way pin operator protects us from SQL injection.

1.4 Query binding

We can refer to table throughout query by using query binding: 

q = from a in "artists", where: a.name == "Bill Evans", select: [:id, :name]

1.5 Composing queries

Instead of writing one big query, we can create it from couple of smaller reusable queries: 

albums_by_miles = from a in "albums", as: :albums,
  join: ar in "artists", as :artists,
  on: a.artist_id == ar.id, where: ar.name == "Miles Davis"

album_query = from [artists: ar, albums, a] in albums_by_miles,
  select: [a.title, ar.name]

We can also use functions to create queries, to make everything cleaner: 

def by_artist(query, artist_name) do
  from a in query,
    join: ar in "artists", on: a.artist_id == ar.id,
    where: ar.name == ^artist_name
end

def with_tracks_longer_than(query, duration) do
  fron a in query,
    join: t in "tracks", on: t.album_id == a.id,
    where: t.duration > ^duration,
    distinct:true
end

def title_only(query) do
  from a in query, select: a.title
end

q = "albums"
|> by_artist("Miles Davis")
|> with_tracks_longer_than(720)
|> title_only

1.6 Combining results

We can combine results of different queries ony if they have results sets with the same column names and data types.

When using for e.g. union:, results will not contain duplicates. But that can have performance impact on database. If we don’t care about duplicates, we can use instead :union_all.

1.7 Schemas

We don’t have to use them everywhere. For e.g. queries for reports shouldn’t require schemas.

1.8 Associations

1.8.1 Nested associations

Let’s say Artist have many Albums, and Album have many Tracks. If we want to access all tracks directly from artist, we would use nested association: 

schema "artists" do
  has_many :albums, MusicDB.Album
  has_many :tracks, through: [:albums, :tracks]
end

But you cannot use belongs_to to access artist from track directly: you have to go track.album.artist.

1.8.2 M:N

If we need to add extra data to linking table, we need to explicitly add schema for it: 

schema "albums" do
  many_to_many :genres, MusicDB.Genre, join_through: MusicDB.AlbumGenre
end

schema "genres" do
  many_to_many :albums, MusicDB.Album, join_through: MusicDB.AlbumGenre
end

schema "albums_genres" do
  # extra fields definitions
  belongs_to :albums, MusicDB.Album
  belongs_to :albums, MusicDB.Genre

Otherwise, we can just use string. 

schema "albums" do
  many_to_many :genres, MusicDB.Genre, join_through: "albums_genre"
end

schema "genres" do
  many_to_many :albums, MusicDB.Album, join_through: "albums_genre"
end

1.9 Changesets

They help us make changes to database in a secure way.

There are 3 phases that are executed by changesets when changes are stored in database:

  1. Casting and filtering
  2. Validation
  3. Sendig data to database and capturing result

1.9.1 Casting and filtering

Here we:

  • filter out which data we want to store
  • cast data (like casting “2018-01-01” to real date)
1.9.1.1 Input data is comming from
1.9.1.1.1 Our own system

If we “own” the data that needs to be inserted, we use change function: 

import Ecto.Changeset
changeset = change(%Artist{name: "James"})
changeset.changes #=> %{name: "James"}
1.9.1.1.2 From outside

If data is comming from “outside” (like from web form), we use cast function: 

params = %{"name" => "Charlie Parker", "birth_date" => "1920-08-29",
           "instrument" => "alto sax"}

changeset = cast(%Artist{}, params, [:name, :birth_date])
changeset.changes
#=> %{birth_date: ~D[1920-08-29], name: "Charlie Parker"}
1.9.1.1.2.1 nil

By default, the cast function will treat the empty string "" as nil when creating the changeset. We can also declare what should be treated as nil: 

params = %{"name" => "Charlie Parker", "birth_date" => "NULL"}
changeset = cast(%Artist{}, params, [:name, :birth_date],
  empty_values: ["", "NULL"])
changeset.changes
#=> %{name: "Charlie Parker"}

1.9.2 Validating

There are 2 types of validation:

  • validations
    • performed without contacting database
    • all validations are ran to collect as much errors
  • constraints
    • used for database constraints (like checking if something is unique)
    • they are executed only when interacting with database
    • will not be reported in chanteset’s errors field

If one constraint fails, the others won’t be checked. If any validations fail, the constraints are not checked at all. Ecto won’t bother trying to send a changeset to the database if it fails validtion.

Problem with constraints is that they are checked after all validations are done. But if we have a long form, and we want to check if username is taken or not, we’ll find out after all validations have been ran. Sometimes we want to check some constraints with validations (and to be shown in errors field). For e.g. we want to check if name is unique. For that we use unsafe_validate_unique.

1.9.3 Sendig data to database and capturing result

This is usual way of sending data to database, and capturing all errors: 

params = %{name: "Gene Harris"}
changeset =
  %Artist{}
  |> cast(params, [:name])
  |> validate_required([:name])

case Repo.insert(changeset) do
  {:ok, artist} -> IO.puts("Record for #{artist.name} was created.")
  {:error, changeset} -> IO.inspect(changeset.errors)
end

Couple of reasons why this works:

  • if changeset sees that there are errors, it won’t even bother with talking to database
  • it’s possible that database will catch a problem with your data that you forgot to write a validation for

1.9.4 Using changeset when not working with database

We can leverage the casting, filtering and validation features of changesets, even if the data is not being persisted in a database. 

params = %{"artist_name" => "Ella Fitzgerald", "album_title" => "",
           "artist_birth_date" => "",  "album_release_date" => "",
           "genre" => ""}

changeset =
{%{}, form}
|> cast(params, Map.keys(form))
|> validate_in_the_past(:artist_birth_date)
|> validate_in_the_past(:album_release_date)

if changeset.valid? do
  # execute the advanced search
else
  # show changeset.errors to the user
end

1.9.5 Associations

1.9.5.1 When we need to add new associated record
artist = Repo.get_by(Artist, name: "Miles Davis")
album = Ecto.build_assoc(artist, :albums, title: "Miles Ahead")
Repo.insert(album)
#=> {:ok, %MusicDB.Album{id: 6, title: "Miles Ahead", artist_id: 1, ...}
1.9.5.2 When working with individual records of association

It’s usually easisest to work with the child record separate from the parent record. build_assoc can help you create new child record for a given parent record, and you can perform updates and deletes by loading the child record and using the update and delete functions provided by Repo.

1.9.5.3 When working with collection at once

You’ll need to think about what you want to do with records that are removed or replaced when the collection is updated (should they be nullified, deleted etc.).

If data is comming from external source, you’ll want to use cast_assoc. Or you can use changesets to create/update/delete the child records on their own, then use put_assoc in a separate changeset to update the collection on the parent record. This is often a great way to work with many-to-many associations.

If data is comming from your application, you’ll want to use put_assoc.

1.11 Repo

Repo is just a simple Elixir module, so we can extend it with functions that would be useful in future use: 

defmodule DottedLine.Repo do
  use Ecto.Repo,
    otp_app: :dotted_line,
    adapter: Ecto.Adapters.Postgres

  def count(table) do
    aggregate(table, :count, :id)
  end
end

2.10 Preloading associations

Two trips to database:

  1. Get all albums
  2. Get all tracks
      albums = Repo.all(from a in Album, preload: :tracks)

One trip to dabase (with limit): 

      q = from a in Album,
        join: t in assoc(a, :tracks),
        where: t.title == "Freddie Freeloader",
        preload: [tracks: t]
