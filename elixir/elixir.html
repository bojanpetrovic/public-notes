1.2 Pattern matching

1.2.1 = match operator

Elixir calls = symbol the match operator.

Joe Armstrong, Erlang’s creator, compares the equals sign in Erlang to that used in algebra. When you write the equation x = a + 1, you are not assigning the value of a + 1 to x. Instead, you’re simply asserting that the expressions x and a + 1 have the same value. If you know the value of x, you can work out the vlaue of a, and vice versa.

1.2.2 _

In some cases, you don’t care about a particular value in a pattern. It is a common practice to bind those values to the underscore, _. The variable _ is special in that it can never be read from. Trying to read from it gives a compile error.

We can also use underscore with variable name to declare that we won’t need it: 

# we don't care about head
def len([_head| tail]), do: 1 + len(tail)

1.2.3 Pin operator ^

If we have 

a = 1
[a,b] = [2, 3]

this will bind 2 to a. But if we want a to force matching, then we use pin operator: 

[^a,b] = [2, 3] # fail, because a=1, so this expression is [1,b] = [2,3]

1.2.4 Recursive pattern matching

Instead of writing something like 

def for_location([ [time, target_loc, temp, rain ] | tail], target_loc) do
  [ [time, target_loc, temp, rain ] | for_location(tail, target_loc)
  end

we can write 

def for_location([ head = [time, target_loc, temp, rain ] | tail], target_loc) do
  [ head | for_location(tail, target_loc)
  end

1.2.5 Can’t bind keys in maps

We can do 

%{ 2 => state} = %{ 1 => :ok, 2 => :error }

but you cannot do this: 

%{ item => :ok} = %{ 1 => :ok, 2 => :error }

In Elixir convention is:

  • double-quoted form is strings

1.4.5.4 Single-Quoted Strings - Character Lists
1.4.5.4.1 What are they

These are called character list in Elixir. They are represented as a list of integer values, each value corresponding to a codepoint in the string.

1.4.5.4.3 ?c notation

The notation ?c returns the integer code for the character c. This is often useful when employing patterns to extract information from character lists. Here’s a simple module that parses the character-list representation of an optionally signed decimal number. 

defmodule Parse do
  def number([ ?- | tail ]), do: _number_digits(tail, 0) * -1
  def number([ ?+ | tail ]), do: _number_digits(tail, 0)
  ...
1.4.5.5 Double-Quoted Strings - Binaries

The contents of a double-quoted string are stored as a consecutive sequence of bytes in UTF-8 encoding.

When Elixir library documentation uses the word string (and most of the time it uses the word binary), it means double-quoted strings.

1.4.9 Collections

1.4.9.1 Lists
1.4.9.1.1 What they can hold?

Multiple values of multiple types.

Values can be non-unique.

1.4.9.1.2 Implementation

Elixir implements list collections as linked lists. This means that accessing the list length is an operation that will run in linear time (O(n)). For this reason, it is typically faster to prepend than to append 

iex> list = [3.14, :pie, "Apple"]
[3.14, :pie, "Apple"]
# Prepending (fast)
iex> ["π"] ++ list
["π", 3.14, :pie, "Apple"]
# Appending (slow)
iex> list ++ ["Cherry"]
[3.14, :pie, "Apple", "Cherry"]
1.4.9.1.3 head|tail operator
1.4.9.1.3.1 Prepending elements to list 
l1 = [1,2,3]
l2 = [0 | l1] # l2=[0,1,2,3]
1.4.9.1.3.2 Extracting head from the rest 
[head|tail]=[1,2,3]
# head: 1
# tail: [2,3]
1.4.9.1.4 Other examples
l = [1,2,3]
[1,2,3] ++ [4,5,6] # concatenation
[1,2,3,4] -- [2,4] #difference
1 in [1,2,3,4] # membership
1.4.9.1.5 Leave off square brackets if list is last argument

Elixir allows us to leave off the square brackets if a keyword list is the last argument in a function call. Thus, 

DB.save record, [ {​:use_transaction​, true}, {​:logging​, ​"​​HIGH"​} ]

can be written more cleanly as 

DB.save record, ​use_transaction:​ true, ​logging:​ ​"​​HIGH"​
1.4.9.2 Tupples
1.4.9.2.1 Implementation

Tuples are similar to lists, but are stored contiguously in memory. This makes accessing their length fast but modification expensive; the new tuple must be copied entirely to memory.

1.4.9.2.2 Example
t = {:ok, 42}
1.4.9.2.3 Common usage

It is common for functions to return a tuple where the first element is the atom :ok if there were no errors.

1.4.9.2.4 List vs Tupple
what pros cons
list   accessing length, or appending to list is linear operation
tupple getting length or element fast updating/adding element expensive (new tuple in memory)

When counting the elements in a data structure, Elixir also abides by a simple rule: the function is named size if the operation is in constant time (i.e. the value is pre-calculated) or length if the operation is linear (i.e. calculating the length gets slower as the input grows). As a mnemonic, both “length” and “linear” start with “l”.

1.4.9.3 Keyword lists
1.4.9.3.1 What it is

Keyword lists and maps are the associative collections of Elixir. In Elixir, a keyword list is a special list of two-element tuples whose first element is an atom; they share performance with lists. 

list = [a: 1, b: 2] # list = [{:a, 1}, {:b, 2}]
1.4.9.3.2 Characteristics
  • Keys must be atoms.
  • Keys are ordered, as specified by the developer.

  • Keys can be given more than once.

    Since it’s a list, they provide the same linear performance characteristics as lists. The longer the list, the longer it will take to find a key, to count the number of items, and so on.

    They are mainly used to pass optional values.

    When the keyword list is the last argument of a function, the square brackets are optional.

    We should use Keyword module to work with keyword lists.

1.4.9.3.3 Where are they used

Keyword lists are most commonly used to pass options to functions.

1.4.9.4 Maps
1.4.9.4.1 Examples
1.4.9.4.1.1 Defining 
states = %{"Al" => "Alabama", "WI" => "Wisconsin"}
states["AL"]

# If atoms are keys
colors = %{red: 0xf00, green: 0x0f0, blue: 0x00f}
colors[:red]
colors.red

# Pattern matching
%{:a => a} = %{:a => 1} # a = 1
1.4.9.4.1.2 Updating key’s value 
map = %{:a =>, 2 => :b}
%{map | 2 => "two"} # %{2 => "two", :a => 1}
1.4.9.4.1.3 Adding key 
Map.put(%{:a => 1, 2 => :b}, :c, 3)
1.4.9.4.1.4 Updating nested structure 
users = [
  john: %{name: "John", age: 27, languages: ["Erlang", "Ruby", "Elixir"]},
  mary: %{name: "Mary", age: 29, languages: ["Elixir", "F#", "Clojure"]}
]

put_in users[:john].age, 31 # updating john's age to 31
1.4.9.4.2 Characteristics
  • they allow any values as a key
  • keys do not follow any ordering
1.4.9.5 Sets

handle_open = fn
  {:ok, file} -> "Read data: #{IO.read(file, :line)}"
  {_, error} -> "Error: #{:file.format_error(error)}"
end

1.5.2.5.2 Optimization

We can write 

speak = fn what -> IO.puts what

as 

speak = &(IO.puts &1)

Normally Elixir would have generated an anonymous function, so &(IO.puts(&1)) would become fn x -> IO.puts(x) end. But Elixir noticed that the body of the anonymous function was simply a call to a named function (the IO function puts) and that the parameters were in the correct order (that is, the first parameter to the anonymous function was the first parameter to the named function, and so on). So Elixir optimized away the anonymous function, replacing it with a direct reference to the function, IO.puts/1. For this to work, the arguments must be in the correct order: 

​ iex>​ rnd = &(Float.round(&1, &2))
 &Float.round/2
 ​ iex>​ rnd = &(Float.round(&2, &1))
 #Function<12.17052888 in :erl_eval.expr/5>

1.5.3.3 Guard clauses
1.5.3.3.1 What it is

They are a way to invoke a function based on argument types, or based on some test involving their value (unlike Pattern matching, where function is invoked based on exact argument passed).

First pattern matching is done, and then guard clauses kicks in.

1.5.3.3.2 Example
defmodule Guard do
  def what_is(x) when is_number(x) do
    IO.puts "#{x} is a number"
  end

  def what_is(x) when is_list(x) do
    IO.puts "#{inspect(x)} is a list"
  end

  def factorial(0), do: 1
  def factorial(n) when is_integer(n) and n>0, do: n * factorial(n-1)
end
1.5.3.4 Default parameters
1.5.3.4.1 Example
defmodule MyModule do
  def myFunc(p1, p2 \\ 2, p3 \\ 3, p4), do: IO.inspect [p1, p2, p3, p4]
end

MyModule.myFunc("a", "b")           # ["a", 2, 3, "b"]
MyModule.myFunc("a", "b", "c")      # ["a", "b", 3, "c"]
MyModule.myFunc("a", "b", "c", "d") # ["a", "b", "c", "d"]
1.5.3.4.2 Default value

Any expression is allowed to serve as a default value, but it won’t be evaluated during the function definition. Every time the function is invoked and any of its default values have to be used, the expression for that default value will be evaluated.

1.5.3.4.3 Function with default values has multiple clauses

If a function with default values has multiple clauses, it is required to create a function head (without an actual body) for declaring defaults: 

defmodule Concat do
  def join(a, b \\ nil, sep \\ " ")

  def join(a, b, _sep) when is_nil(b) do
    a
  end

  def join(a, b, sep) do
    a <> sep <> b
  end
end
1.5.3.5 Private functions (defp)

They can be called only within the module that declares it: 

defmodule MyModule do
  defp myPrivateFunc(n), do: IO.puts n
end
1.5.3.6 do end
def double(n) do
  n*2
end

can be shortened into:

1.5.4 Parantheses

We can omit them when declaring or invoking function. But if function has no argument, we must use them when invoking function.

You should always use parentheses around function parameters in pipelines.

1.6.3 cond

1.6.3.1 Example
cond do
  2 + 2 == 5 ->
    "This is never true"
  2 * 2 == 3 ->
    "Nor this"
  true ->
    "This is always true (equivalent to else)"
end
1.6.3.2 When to use

To check different conditions and find the first one that evaluates to true. It’s basically else if clauses in many imperative languages.

1.6.4 with

1.6.4.1 Doesn’t change outer scope
content = "ok"
with {:ok, file} = File.open("with.esx"),
     content     = IO.read(file, :all),
       :ok       = File.close(file)
  do
  IO.puts(content) # what was read from file
end

  IO.puts content # "ok"

This code didn’t changed outer content variable.

1.6.4.2 with is treated as a function

Underneath the covers, with is treated by Elixir as if it were a call to a function or macro. That means that you cannot write this: 

mean = with # WRONG!
   count = Enum.count(values),
     sum   = Enum.sum(values)
     do
     sum/count
   end

Instead, you can put the first parameter on the same line: 

mean = with count = Enum.count(values),
            sum   = Enum.sum(values)
  do
  sum/count
end

or 

mean = with(
         count = Enum.count(values),
         sum   = Enum.sum(values)
           do
           sum/count
         end)
1.6.4.3 else

with supports else: 

import Integer

m = %{a: 1, c: 3}

a =
  with {:ok, number} <- Map.fetch(m, :a),
       true <- is_even(number) do
    IO.puts "#{number} divided by 2 is #{div(number, 2)}"
    :even
  else
    :error ->
      IO.puts("We don't have this item in map")
    :error

    _ ->
      IO.puts("It is odd")
    :odd
  end
1.6.4.4 Using <- instead of =

If you use <- instead of = in a with expression, it performs a match, but if it fails it returns the value that couldn’t be matched: 

​ with​ [a|_] <- [1,2,3], ​do​: a # 1
  with​ [a|_] <- nil,     ​do​: a # nil

This can be used for e.g. in a code where we want to return nil if the user can’t be found: 

person = with [_, uid, gid] <- Regex.run(..., content)
  do
  "group :#{gid}, user: #{uid}"
end

If regex doesn’t return something that can be matched, person will be nil.

1.6.4.5 Example of when with simplifes things

Without 

case Repo.insert(changeset) do
  {:ok, user} ->
    case Guardian.encode_and_sign(user, :token, claims) do
      {:ok, token, full_claims} ->
        important_stuff(token, full_claims)

      error ->
        error
    end

  error ->
    error
end

With 

with {:ok, user} <- Repo.insert(changeset),
     {:ok, token, full_claims} <- Guardian.encode_and_sign(user, :token, claims) do
  important_stuff(token, full_claims)
end

1.8 Modules

1.8.1 Referencing functions

Functions inside the same module can call each other without module name.

1.8.2 Module nesting

Module nesting in Elixir is an illusion - all modules are defined at the top elvel. When we define a module inside another, Elixir simply prepends the outer module name to the inner module name, putting a dot between the two.

There’s no particular relationship between nested modules.

1.8.3.3 require

We could use require to tell Elixir you’re going to use macros from other module. The slight difference with import is that it allows using macros, but not functions from the specified module.

1.8.3.4 use

With the use macro we can enable another module to modify our current module’s definition.

  defmacro __using__(_opts) do
    quote do
      def hello(name), do: "Hi, #{name}"
    end
  end
end

defmodule Example do
  use Hello
end

Example.hello("Sean")

1.8.4 Module attributes

Elixir modules each have associated metadata.

We use them mostly for configuration and metadata only (like Java programmers might use constants)

You can’t set an attribute inside a function definition, only at the top level of a module. 

defmodule MyModule do
  @author "Bojan"
  def get_author, do: @author
end

1.8.5 Module names

Module names are just atoms. When you write a name starting with an uppercase letter, such as IO, Elixir converts it internally into an atom of the same name with Elixir. prepended. So, IO becomse Elixir.IO, and Dog becomes Elixir.Dog.

When we’re calling stuff from Erlang, we use colon: 

:io.format("The number is ~3.1f~n", [5.678])

1.13 Concurrency

1.13.1 Actor model of concurrency

An actor is an independent process that shares nothing with any other process. You can spawn new processes, send them messages, and receive messages back.

1.13.2 Linking processes

We can link 2 processes, so that they are aware if other process dies. If other process dies, all related processes die by default.

1.13.3 Montiroing processes

Unlike with linked processes, one process can monitor other process, and be notified if monitored process ended. But, unlike with linked processes, monitored process doesn’t receive termination signal of monitoring process.

1.13.4 Parallel Map

It’s like map, but it applies the function to each element in a separate process.

1.15 Elixir internals

1.15.1 Performance

1.15.1.1 Immutability

We don’t change list when we add new element: we create new one. This will not create new list in memory. Since everything is immutable in Elixir, new list will be created by resuing old list, and appending new element.

1.15.1.2 Garbage collection

All code runs in processes. Processes are isolated from each other. Each has it’s own heap. That’s why garbage collection runs faster. For e.g. if process terminates, there’s no garbage collection required.

Atoms are not garbage collected, to don’t use them when returning results from database.

1.15.1.3 Tail-call optimization

When doing multiple recursive calls, we’re not producing multiple stacks. This doesn’t happen in Elixir, as it implements tail-call optimization. If the last thing a function does is call itself, there’s no need to make the call. Instead, the runtime simply jumps back to the start of the function. If the recursive call has arguments, then these replace the original parameters. But beware—the recursive call must be the very last thing executed.

This doesn’t have tail-call optimization: 

def factorial(0), do: 1
def factorial(n), do: n * factorial(n-1)

But we can make it happen like this: 

defmodule TailRecursive do
  def factorial(n), do: _fact(n, 1)
  defp _fact(0, acc), do: acc
  defp _fact(n, acc), do: _fact(n-1, acc*n)
end
